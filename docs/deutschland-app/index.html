Deutschland-App: Die Verwaltungs-Super-App

Alle staatlichen Dienste in einer App – ohne App-Sprünge, ohne Multi-Login. eID, Wallet, Anträge, Postfach, Consent, Behördenkommunikation, Payment und Statusverfolgung in einer einzigen, nutzerzentrierten Super-App.

1 · Super-App vs. Multi-App – Ist das sinnvoll?

Bewertung

Ja – eine Super-App ist sinnvoll, realistisch und international erprobt. Der Schlüssel liegt in der richtigen Architektur: App-Shell + dezentrale Module + verbindliche APIs.

3 · Dezentrale Umsetzungsarchitektur

Das Kernproblem
Wie baut man eine einheitliche App, wenn 50+ Teams an 17+ Building Blocks arbeiten, jeder sein eigenes Release-Tempo hat und niemand auf niemanden warten soll?
Lösung

App-Shell + Module Federation + API-Integration. Die App-Shell ist der einheitliche Container; Module werden von verschiedenen Teams gebaut und zur Laufzeit integriert.

App-Architektur (Schichtenmodell)

App-Shell (zentral gepflegt)
Navigation & Tab-Bar Auth-Gate (Biometrie) Design-System Push-Handler Offline-Layer Module-Loader Shared APIs (Consent, Payment, Signatur)
Funktionsmodule (dezentral gepflegt)
eID / Auth (BB-01) Wallet (BB-05) Formular-Engine (BB-02) Postfach (BB-03) Payment (BB-04) E-Signatur (BB-10) KI-Chatbot (BB-12) Leistungskatalog (BB-13/17) Termine (BB-15) Consent-Center (BB-09)
On-Device Plattform
Secure Enclave SQLite / Realm Biometrie-API Kryptografie FCM / APNs
⇄ Backend APIs (via API-Gateway BB-07)
BB-01 … BB-17 APIs OpenAPI 3.1 mTLS OIDC Token

Drei Integrationsmuster

Muster 1: Native Module

Für Hardware-nahe Features

Einsatz: Wallet, eID/PID, Biometrie, E-Signatur, Payment, Push
  • Module werden als native Libraries (Swift Package / Kotlin Module) gebaut
  • App-Shell bindet sie über ein Module Interface Protocol ein
  • Nutzen zentrale Design-System-Library
  • Updates über App Store Release (2-Wochen-Zyklus)
Muster 2: WebView-Module

Für Content- und Formular-Features

Einsatz: Formulare, Leistungskatalog, Fachportale, KI-Chatbot
  • Module werden als Micro-Frontends (React/Vue) auf CDN deployt
  • App lädt sie in einen eingebetteten WebView
  • JavaScript Bridge für native Features: Consent-Dialog, Signatur, Payment
  • Module-Teams können täglich deployen ohne App-Update
Muster 3: API-only

Für Daten-Features

Einsatz: Postfach-Listen, Antragsstatus, Termine, Consent-Dashboard, Protokolle
  • App-Shell enthält die UI-Komponenten (Listen, Karten, Detail-Views)
  • BB-Teams liefern nur die API (OpenAPI 3.1 Contract)
  • Lokales Caching für Offline-Zugang
  • Kein eigener UI-Code vom BB-Team nötig

4 · Module Interface Contract

Damit dezentrale Teams Module liefern können, braucht es einen verbindlichen Vertrag:

interface DeutschlandAppModule {
  // Identifikation
  moduleId: string; // z.B. "postfach", "wallet"
  version: string; // SemVer
  integrationType: 'native' | 'webview' | 'api-only';

  // Lifecycle
  initialize(ctx: AppContext): Promise<void>;
  activate(): void;
  deactivate(): void;

  // Navigation & Deep Links
  routes: RouteDefinition[];
  deepLinks: DeepLinkPattern[]; // "deutschland://postfach/{id}"

  // Badges & Offline
  getBadgeCount(): Promise<number>;
  offlineCapable: boolean;
}

// Die App-Shell stellt bereit:
interface AppContext {
  authToken: string; // OIDC Access Token
  theme: ThemeTokens; // Design-System Tokens
  bridge: AppBridge; // Consent, Payment, Signatur
  apiGateway: string; // API-Gateway Base URL
}

5 · Governance-Modell

Damit dezentrale Entwicklung nicht in UX-Chaos mündet:

1 · Modul-Zulassung: Jedes Modul durchläuft UX-, Security-, Performance- und A11y-Review
2 · Design-System-Pflicht: Module MÜSSEN die zentrale Component Library nutzen – keine Custom-Designs
3 · Contract-First: Interface Contract wird vor der Entwicklung definiert und abgenommen
4 · Automatisierte Quality Gates: CI/CD prüft: Design-Conformance (Screenshot-Diff), WCAG-Scan, Ladezeit < 2s, SAST
5 · Release-Entkopplung: WebView-Module deployen unabhängig; Native Module im 2-Wochen-Zyklus
Zentrales Team Verantwortung
Shell-Team App-Shell, CI/CD, Module-Loader, Infrastruktur
UX-Team Design-System, Barrierefreiheit, UX-Reviews, Quality Gates
Security-Team Pen-Tests, Code-Reviews, Dependency-Scanning
Modul-Teams (dezentral) Entwickeln, testen, deployen eigenständig nach Contract
