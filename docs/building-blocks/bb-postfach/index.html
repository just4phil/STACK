BB-03 · Postfach & Kommunikation

Föderale Kommunikationsinfrastruktur für die sichere, E2E-verschlüsselte Kommunikation zwischen Bürger:innen, Organisationen und öffentlichen Stellen – basierend auf Matrix und MLS (RFC 9420), gemäß ZAPUK-Zielarchitektur des IT-Planungsrats.

§2 Beschreibung

Zweck

BB-03 Postfach spezifiziert eine einheitliche föderale Kommunikationsinfrastruktur, die sämtliche bestehenden Postfach- und Kommunikationslösungen der öffentlichen Hand konsolidiert. Die Infrastruktur ist eine fachlich agnostische Querschnittskomponente (ADR-0001) und ersetzt den fragmentierten Ist-Zustand aus ZBP, MJP, OZG-Plus-Postfach, Postfach 2.0, De-Mail, EGVP und weiteren Insellösungen.

Topologie

Offene, föderierte Client-Server-Architektur mit hybrider Topologie (ADR-0002): Ein zentral bereitgestelltes Backend für alle Nutzergruppen (ADR-0006) sowie die Möglichkeit für Behörden, Unternehmen und andere Organisationen, eigene Backends nach Registrierung und Einhaltung der Anschlussbedingungen zu betreiben (ADR-0004).

§3 Begriffe & Definitionen

Begriff Definition
Matrix Offener Standard für dezentrale Echtzeitkommunikation (spec.matrix.org). Server replizieren strukturierte Nachrichten über Föderations-API.
MLS Messaging Layer Security (RFC 9420). IETF-standardisiertes E2E-Verschlüsselungsprotokoll mit PFS, Post-Compromise Security, effizienter Gruppenverschlüsselung.
Homeserver Matrix-Server, der Nutzerkonten verwaltet, Nachrichten speichert (verschlüsselt) und mit anderen Homeservern föderiert.
Postfachzugang Client-Anwendung (Web, App, Fachverfahren), die über die Matrix Client-Server API mit dem Homeserver kommuniziert.
Postfach-Backend Serverseitige Komponente (Matrix Homeserver + Erweiterungen) für Nachrichtenspeicherung, Federation, Zustellbestätigung.
Room Matrix-Raum: Kommunikationskanal zwischen zwei oder mehr Teilnehmenden. Enthält Events (Nachrichten, State-Changes).
Event Atomare Einheit im Matrix-Protokoll: Nachricht, Statusänderung oder Systeminformation innerhalb eines Rooms.
MLS Group Kryptografische Gruppe im MLS-Protokoll. Bildet den E2E-verschlüsselten Kanal, der einem Matrix-Room zugeordnet wird.
Self-Service-Portal Zentrale Verwaltungsoberfläche für die Registrierung von Backends, Zustimmung zu Anschlussbedingungen und SDK-Download (ADR-0008).
SDK Software Development Kit zur vereinfachten Anbindung von Postfachzugängen inkl. MLS-Verschlüsselung (ADR-0009).
Zero-Trust Sicherheitsarchitektur: Kein implizites Vertrauen in Netzwerk oder Infrastruktur. Authentizität wird auf E2E-Ebene sichergestellt.
Anschlussbedingungen Technische und organisatorische Voraussetzungen für den Betrieb dezentraler Backends (ADR-0004).
Federation Verbindung mehrerer Homeserver zum Kommunikationsverbund über Matrix Server-Server API.

§5 Querschnittsanforderungen

5.1 Sicherheit

ID Anforderung Referenz
SEC-01 Alle Nachrichten und Anhänge MÜSSEN E2E-verschlüsselt übertragen werden. Kein Aufbrechen der Verschlüsselung durch Backends. ADR-0012, RFC 9420
SEC-02 MLS MUSS als primäres E2E-Protokoll implementiert werden. Olm/Megolm ist als Übergangslösung zulässig. ADR-0012
SEC-03 Perfect Forward Secrecy (PFS) und Post-Compromise Security (PCS) MÜSSEN unterstützt werden. RFC 9420 §16
SEC-04 Zero-Trust: Authentizität MUSS auf der E2E-Verschlüsselungsschicht hergestellt werden, nicht auf Transportebene. ADR-0013
SEC-05 Transport-Verschlüsselung: TLS 1.3+ für alle Server-zu-Server und Client-zu-Server Verbindungen. BSI TR-02102-2
SEC-06 Key-Management: MLS KeyPackage Distribution, automatische Key-Rotation, Geräteverifikation via Cross-Signing. RFC 9420 §11
SEC-07 Kryptographische Algorithmen MÜSSEN den BSI-Vorgaben entsprechen (TR-02102-1). BSI TR-02102-1

5.2 Datenschutz

ID Anforderung Referenz
PRI-01 Metadaten-Minimierung: Backends speichern nur die für Federation notwendigen Metadaten. Nachrichteninhalte sind serverseitig nicht einsehbar. DSGVO Art. 25
PRI-02 Löschfristen: Nutzer:innen MÜSSEN Nachrichten endgültig löschen können. Konfigurierbare Aufbewahrungsfristen. DSGVO Art. 17
PRI-03 Datenportabilität: Export aller eigenen Nachrichten in offenem Format MUSS möglich sein. DSGVO Art. 20
PRI-04 Kein Tracking: Keine Analyse von Kommunikationsverhalten durch Backends. DSGVO Art. 5

5.3 SLA & Performance

§6 Funktionale Anforderungen

6.1 Messaging (FA-MSG)

ID Prio Anforderung
FA-MSG-01 MUSS Senden und Empfangen von Textnachrichten über Matrix Client-Server API (m.room.message).
FA-MSG-02 MUSS Senden und Empfangen von Anhängen (Dateien, Bilder, Dokumente) über Content Repository API mit E2E-Verschlüsselung.
FA-MSG-03 MUSS Zustellbestätigung: Empfänger-Client sendet automatisch eine Empfangsbestätigung (m.receipt).
FA-MSG-04 MUSS Asynchrone Kommunikation: Nachrichten werden serverseitig gespeichert, bis alle Geräte des Empfängers synchronisiert haben.
FA-MSG-05 SOLL Lesebestätigung: Optional konfigurierbar je Nutzer:in.
FA-MSG-06 SOLL Strukturierte Nachrichtentypen: Terminabfragen, Abstimmungen, Formulare über Custom Matrix Event Types.
FA-MSG-07 SOLL Nachrichtenantworten (Threading) und Referenzierung (m.relates_to).

6.2 Föderation (FA-FED)

ID Prio Anforderung
FA-FED-01 MUSS Kommunikation zwischen Nutzer:innen auf unterschiedlichen Homeservern über Matrix Server-Server API.
FA-FED-02 MUSS DNS-basiertes Server-Discovery (.well-known/matrix/server).
FA-FED-03 MUSS Backend-Registrierung über Self-Service-Portal mit digitaler Zustimmung zu Anschlussbedingungen.
FA-FED-04 MUSS Möglichkeit des Ausschlusses eines Backends bei Verstoß gegen Anschlussbedingungen.
FA-FED-05 SOLL Interoperabilität mit EU-Systemen unter Aufrechterhaltung der E2E-Verschlüsselung (ADR-0010).

6.3 E2E-Verschlüsselung (FA-E2E)

ID Prio Anforderung
FA-E2E-01 MUSS MLS Group-Erstellung für jeden Kommunikationsraum. Schlüsselmaterial wird per MLS Delivery Service verteilt.
FA-E2E-02 MUSS Automatische Key-Rotation gemäß MLS-Epoch-Mechanismus.
FA-E2E-03 MUSS Multi-Device: Jedes Gerät hat eigenes MLS-Keypair. Hinzufügen/Entfernen von Geräten aktualisiert die MLS-Gruppe.
FA-E2E-04 MUSS Kryptographische Identitätsprüfung der Kommunikationspartner auf E2E-Ebene (ADR-0013).
FA-E2E-05 SOLL Olm/Megolm-Fallback für Matrix-Clients, die MLS noch nicht unterstützen (Übergangsphase).
FA-E2E-06 KANN Post-Quantum-Sicherheit: Vorbereitung für PQ-Cipher-Suites in MLS, sobald IETF-standardisiert.

6.4 Fachverfahren & Adressierung (FA-FV)

ID Prio Anforderung
FA-FV-01 MUSS SDK für Fachverfahren-Integration: Automatisierter Nachrichtenversand (Bescheide, Statusupdates) inkl. MLS-Verschlüsselung.
FA-FV-02 MUSS Adressierung über Matrix User-IDs (@nutzer:homeserver.bund.de). Verzeichnisdienst für Auflösung.
FA-FV-03 SOLL QR-Code- und Deep-Link-basierte Kontaktaufnahme für Bürger:innen.
FA-FV-04 SOLL Bidirektionale Kommunikation: Rückkanal für Bürger:innen-Rückfragen an Behörden.
FA-FV-05 KANN Qualifizierte Elektronische Signatur (QES) auf Nachrichtenebene für rechtsverbindliche Nichtabstreitbarkeit (ADR-0017).

§7 Datenmodell

7.1 Matrix Event-Struktur

Nachrichten werden als Matrix Events übertragen. Die Payload ist E2E-verschlüsselt (MLS-Ciphertext).

{
  "type": "m.room.encrypted",
  "room_id": "!room:homeserver.bund.de",
  "sender": "@absender:homeserver.bund.de",
  "content": {
    "algorithm": "m.mls.v1",
    "ciphertext": "<MLS-verschlüsselter Payload>",
    "session_id": "<MLS Group ID>",
    "device_id": "DEVICE_A"
  },
  "origin_server_ts": 1740000000000,
  "event_id": "$eventId"
}

// Entschlüsselter Payload (nur im Client sichtbar):
{
  "type": "m.room.message",
  "content": {
    "msgtype": "m.text",
    "body": "Ihr Antrag wurde bewilligt.",
    "de.bund.postfach": {
      "message_class": "bescheid",
      "aktenzeichen": "AZ-2026-12345",
      "reply_allowed": true,
      "retention_days": 365
    }
  }
}

9.3 Legacy-Migration

Migrationsstrategie
Bestehende Postfachlösungen (ZBP, MJP, OZG-Plus-Postfach, EGVP, De-Mail) werden über zeitlich befristete Bridge-Gateways angebunden, die Nachrichten aus Legacy-Protokollen (OSCI, REST/proprietär, SMTP/S-MIME) in Matrix Events transformieren. Ziel: Vollständige Ablösung der Legacy-Systeme innerhalb von 36 Monaten nach Produktivstart der ZAPUK-Infrastruktur.
